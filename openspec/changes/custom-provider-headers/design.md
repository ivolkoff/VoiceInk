## Context

Пользователи могут подключить кастомного LLM-провайдера в VoiceInk, выбрав "Custom" в `APIKeyManagementView`. При этом они указывают:
- API Endpoint URL
- Model Name
- API Key (хранится в Keychain, передаётся как `Authorization: Bearer`)

Ряд провайдеров (self-hosted proxy, Cloudflare Workers AI, Azure OpenAI, специфичные on-prem решения) требуют дополнительные HTTP-заголовки для аутентификации, роутинга или кастомной конфигурации.

Текущий вызов `OpenAILLMClient.chatCompletion()` устанавливает только два заголовка: `Content-Type` и `Authorization`. Параметр `extraBody` уже существует для кастомных body-параметров, но для заголовков аналогичного механизма нет.

LLMkit — внешний Swift Package (https://github.com/Beingpax/LLMkit.git), принадлежащий тому же разработчику. Его можно модифицировать через PR в основной репозиторий или через локальный оверрайд.

## Goals

**Goals:**
- Дать пользователю возможность добавлять произвольные HTTP-заголовки (ключ + значение) для кастомного провайдера
- Заголовки сохраняются между запусками приложения (UserDefaults)
- Заголовки передаются в каждом запросе к кастомному провайдеру
- UI для управления заголовками: добавление, удаление, редактирование
- Заголовки включаются в экспорт/импорт бэкапа
- Поддержка множественных заголовков (0 или более пар ключ-значение)

**Non-Goals:**
- Не добавлять заголовки для встроенных провайдеров (OpenAI, Anthropic, Gemini и т.д.)
- Не добавлять валидацию заголовков на уровне приложения (провайдер сам отвечает за корректность)
- Не затрагивать transcription-сервисы — только LLM enhancement flow

## Decisions

### 1. Формат хранения: `[String: String]` в UserDefaults
- **Релизация**: AIService получает свойство `@Published var customHeaders: [String: String]`, которое синхронизируется с UserDefaults
- **Ключ UserDefaults**: `"customProviderHeaders"`
- **Кодирование**: JSONEncoder/JSONDecoder (словарь [String: String] напрямую)
- **Альтернатива**: Хранить в отдельной модели — избыточно для пар ключ-значение

### 2. Добавление custom headers в OpenAILLMClient
- **Решение**: Добавить опциональный параметр `extraHeaders: [String: String]?` в `OpenAILLMClient.chatCompletion()`
- **Логика**: В начале метода, после установки базовых заголовков (`Content-Type`, `Authorization`), итерировать `extraHeaders` и вызывать `request.setValue(value, forHTTPHeaderField: key)`
- **Почему не через существующие механизмы**: 
  - `extraBody` влияет на тело запроса, не на заголовки
  - URLSession не имеет глобального механизма добавления кастомных заголовков
  - Создание прокси-слоя поверх OpenAILLMClient дублирует код
- **Процесс**: LLMkit — репозиторий того же разработчика; предложить изменения через PR или включить локальную копию

### 3. UI: Inline-список заголовков в APIKeyManagementView
- **Место**: В секции `.custom` провайдера, под Model Name и API Key
- **UX**:
  - Заголовки отображаются как список строк "Key: Value" с кнопкой удаления (×)
  - Кнопка "+ Add Header" добавляет новую пару
  - Каждая пара редактируется в TextField'ах
  - Пустые ключи не сохраняются (валидация на уровне модели)
- **Альтернатива**: Popover или Sheet — избыточно для простого списка пар ключ-значение

### 4. Проброс через AIEnhancementService
- В методе `makeRequest()` для кастомного провайдера, при вызове `OpenAILLMClient.chatCompletion()`, передавать `aiService.customHeaders` как `extraHeaders`
- Текущий код (default-кейс) уже обрабатывает Anthropic отдельно, остальные (включая custom) идут через OpenAILLMClient

### 5. Бэкап
- Добавить поле `customProviderHeaders: [String: String]?` в `GeneralBackup`
- Добавить ключ `"customProviderHeaders"` в импортёр
- **Почему в GeneralBackup**: Это настройка AI провайдера, не модель транскрипции

## Risks / Trade-offs

- **[Risk] Модификация внешнего пакета LLMkit**: Если PR не будет принят, нужно будет подключить форк или локальную копию → **Mitigation**: Убедиться, что изменения обратно совместимы (параметр опциональный)
- **[Risk] Безопасность заголовков**: Пользователь может сохранить чувствительные данные (токены) в заголовках → **Mitigation**: Хранение в UserDefaults (не Keychain) — осознанный выбор пользователя; добавляем дисклеймер. При бэкапе экспортируются как есть
- **[Risk] Конфликт с Authorization**: Если пользователь добавит заголовок `Authorization`, он перезапишет стандартный → **Mitigation**: Добавить предупреждение в UI, если ключ заголовка совпадает с зарезервированным (`Authorization`, `Content-Type`)
